# Steer — Roadmap de planes de implementación

El diseño (`docs/design.md`) se implementa como una secuencia de planes, uno por hito
funcional. Cada plan produce software funcional y testeable por sí mismo. Los planes se
ejecutan en orden porque cada uno depende de los tipos/interfaces que fija el anterior.

| # | Plan | Entrega | Depende de |
|---|---|---|---|
| 01 | **Cimientos** (`2026-06-15-01-foundation.md`) | Módulo Go, config-driven (`steer.toml`), sesión AWS, interfaces de capacidad, esqueleto CLI (Cobra) con `-e/--env` + guard de prod + `steer config init\|validate`, paquete `render`, CI. | — |
| 02 | **Capacidad `service`** | `service status\|ls\|deploy\|redeploy\|rollback\|scale\|promote\|watch`, deploy interactivo (picker fuzzy + tag + progreso). Vertical slice que valida toda la arquitectura. **Prioriza el flujo "sin miedo": picker → tag → preview de qué va a pasar → confirmar → progreso claro → rollback a un comando.** | 01 |
| 03 | **Capacidad `registry`** | `registry ls\|images\|build\|prune`. | 01, 02 |
| 04 | **Capacidad `db`** | `db status\|slow-queries\|tunnel`. | 01 |
| 05 | **Capacidades menores** | `queue ls\|watch`, `storage ls`, `host ls\|connect`, `env ls\|up\|down`, `assets deploy\|url\|info\|invalidate`. | 01 |
| 06 | **TUI híbrida** | `steer tui`: dashboard de salud + paleta ⌘k + drill-down, sobre las capacidades ya construidas. | 01–05 |
| 07 | **Distribución** | GoReleaser, Homebrew tap, release por tag. | 01 |
| 08 | **Onboarding + errores que enseñan** | `config init` tipo wizard (detecta perfiles AWS, valida, errores amables), defaults sensatos, mensajes de error "falló X porque Y, prueba Z". Deriva del target (devs que no dominan cloud). | 01 |

**Estado:** Plan 01 escrito. Los planes 02–08 se escriben a medida que se cierra el anterior
(para mantener firmas e interfaces consistentes).
